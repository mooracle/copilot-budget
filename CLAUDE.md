# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**Copilot Budget** is a VS Code extension that tracks GitHub Copilot token usage and measured cost in AI Credits (AIC; 1 AIC = $0.01) across sessions and optionally appends AI budget git trailers to commit messages. It activates on startup, restores stats from any previous session, polls Copilot Chat's OTel span database every 30 seconds, and displays cost in the status bar. Cost is derived from server-reported token counts (measured per-span) and a per-model rate card mirrored from `github/docs:data/tables/copilot/models-and-pricing.yml`.

**OTel-only design.** Cost is always sourced from Copilot Chat's OTel SQLite store (`<globalStorage>/github.copilot-chat/agent-traces.db`) via `node:sqlite`. On first activation in a workspace where the upstream `github.copilot.chat.otel.dbSpanExporter.enabled` setting is unset at both Global and Workspace scope, the extension writes `true` at Workspace scope (`autoEnableOTel` in `config.ts`). An explicit user choice in either scope is preserved — the write is strictly asymmetric: only unset → `true`, never any other transition. After auto-enable, Copilot Chat may need a reload before it starts writing spans; until the DB appears, the status bar renders a clickable `$(refresh) Copilot Budget — reload to start tracking` nudge bound to `workbench.action.reloadWindow`. There is no Files mode fallback, no JSONL parsing, no tilde decoration — measured tokens are the only source.

## Build & Development Commands

```bash
npm run compile          # Build: src/extension.ts → dist/extension.js (esbuild)
npm run watch            # Watch mode with auto-rebuild
npm run lint             # ESLint on src/**/*.ts
npm test                 # Run all Jest tests
npm test -- --watch      # Jest watch mode
npm test -- <pattern>    # Run single test file (e.g. npm test -- tracker)
npm run package          # Create .vsix via @vscode/vsce
npm run update-rates     # Refresh data/models-and-pricing.yml from github/docs upstream
```

The rate card (`data/models-and-pricing.yml`) is a byte-identical mirror of `github/docs:data/tables/copilot/models-and-pricing.yml`. When GitHub publishes new pricing, run `npm run update-rates` and commit the updated YAML alongside the diff so reviewers see the change directly.

Debug: Press F5 in VS Code to launch Extension Development Host.

## Architecture

The extension follows a **baseline/delta** model: at activation it snapshots the maximum `end_time_ms` from `agent-traces.db` as a high-water mark, then on every scan it asks the reader for per-model token sums of all chat spans with `end_time_ms > baseline` that belong to the current window's session ids. The delta against the previous snapshot is what gets reported.

**Entry point**: `src/extension.ts` — `activate(context)` branches on `context.storageUri`. When defined (workspace open) it `await autoEnableOTel()`, constructs an `OTelReader` rooted at `context.globalStorageUri`, builds a session-id provider from `discoverSessionIds(context.storageUri)`, instantiates `Tracker(reader, sessionIdsFn)`, restores previous-session stats from the tracking file (if present), wires up the status bar (including a one-shot nudge if `isOTelDbExporterEnabled() && !reader.isAvailable()`), registers commands (including `showStats`/Open Panel + `showDiagnostics`), optionally auto-installs the git hook, and subscribes to `onConfigChanged` for re-running the hook installer when settings flip. No mode-swap, no `pickSource`, no `swapSource`. When `storageUri` is undefined (empty window) it skips tracker/tracking-file/hook setup entirely, renders a static `$(circle-slash) Copilot Budget` status bar bound to `showDiagnostics`, and registers the other commands as info-message handlers.

**Core modules** (all in `src/`):

- **tracker.ts** — Central state machine. `Tracker` is constructed with an `OTelReader` and a `() => string[]` session-id provider. Polls every 30s via `setInterval`, emits `onStatsChanged` events. Returns `TrackingStats` with per-model `{inputTokens, outputTokens, cacheReadTokens, cacheCreationTokens, costAic}` plus totals `{since, lastUpdated, totalTokens, interactions, totalAiCredits}`. There is no `mode` field anywhere in the stats. `scan()` accumulates session ids into a sticky `seenSessionIds` Set (so a session aging out of discovery doesn't shrink the reported delta), calls `reader.aggregateSince(baselineMs, ids)`, and converts the rows into a per-model token snapshot via `rowsToSnapshot`. `rowsToSnapshot` strips Copilot's `copilot/` / `copilotcli/` / `claude-code/` request-routing prefixes, normalizes the id via `getRateCard`, and subtracts `cacheReadTokens + cacheCreationTokens` from the OTel `input_tokens` value (which is the full prompt token count per OTel GenAI semantics) before pricing — clamped at 0 in case the provider mis-reports. `setPreviousStats(restored)` merges a prior session's stats into the current delta; `consume()` rebases the baseline to `lastSnapshot` so the next commit only attributes post-truncation activity. `dispose()` calls `reader.close()` and stops the timer.
- **sessionDiscovery.ts** — Discovers Copilot session ids for the current window only via `discoverSessionIds(storageUri): string[]`. Primary scan: `<workspaceStorage>/<hash>/GitHub.copilot-chat/transcripts/` (the path used by current Copilot Chat builds, e.g. in devcontainers). Legacy scan: `<workspaceStorage>/<hash>/chatSessions/` (older versions). Both directories are scanned and merged with stem-based dedup; non-session names (matching `NON_SESSION_PATTERNS`) and zero-byte files are filtered, and an mtime cutoff drives `copilot-budget.sessionMaxAgeDays`. Returns UUID stems (`.jsonl` stripped) — these become the SQL placeholders for the OTel session-id filter. When `storageUri` is undefined returns `[]`. Exports `getDiscoveryDiagnostics(storageUri)` returning `{platform, homedir, storageUri, transcriptsDir, legacyChatSessionsDir, filesFound}` — `storageUri` / `transcriptsDir` / `legacyChatSessionsDir` are all `null` in the empty-window state.
- **otelReader.ts** — Opens `<copilot-chat globalStorage>/agent-traces.db` readonly via `node:sqlite` (`new DatabaseSync(path, { readOnly: true })`). Exports `PerModelAggregate` type, `OTelReader` interface (`isAvailable()`, `aggregateSince(sinceMs, sessionIds): PerModelAggregate[]`, `getLatestTimestamp(): number`, `close()`), `createOTelReader(globalStorageUri)`, and `resolveOTelDbUri(globalStorageUri)`. `aggregateSince` returns per-model sums via a single grouped SQL query: `WHERE operation_name = 'chat' AND end_time_ms > ?` (strict — pairs with `MAX(end_time_ms)` as the high-water mark) AND `chat_session_id IN (?, ?, ...) OR EXISTS (... copilot_chat.parent_chat_session_id IN (?, ?, ...))`. The OR-on-parent join is what captures Copilot's background title-subagent spans (gpt-4o-mini), which carry only `parent_chat_session_id`. A LEFT JOIN to `span_attributes` recovers `gen_ai.usage.cache_creation.input_tokens` per span (older spans may lack it). Zero session ids returns `[]` without touching the DB. The `toFiniteInt` helper coerces null/bigint values to safe finite numbers. Schema reference (`spans` denormalized columns + `span_attributes`) is documented at the top of the file. The DB handle is owned by the reader instance; `Tracker.dispose()` calls `reader.close()` so the handle is released on deactivation.
- **tokenRates.ts** — Loads and normalizes the per-model rate card from `dist/models-and-pricing.json` at first use (lazy `fs.readFileSync` + `JSON.parse`). The JSON is generated at build time from `data/models-and-pricing.yml` (byte-identical mirror of upstream) by `esbuild.js`, so the runtime bundle has no `js-yaml` dependency. At load time each rate field is multiplied by 100 so stored rates are AIC per 1M tokens (the source YAML on disk stays USD). Exports `RateCard` type, `getRateCard(modelId)` (strips `copilot/`, `copilotcli/`, `claude-code/` prefixes, then exact match against normalized ids — no family fallback), `computeCost(modelId, tokens)` returning AIC per `(input × rate.input + cacheRead × rate.cachedInput + cacheCreation × (rate.cacheCreation ?? rate.input) + output × rate.output) / 1e6`, `getDisplayName(modelId)`, `stripModelPrefix(modelId)`, and `normalizeModelId(rawName)`. Unknown models return `null`/`0` to avoid silent mispricing.
- **amountFormatter.ts** — Pure formatting primitive used by status bar, tooltip, and panel. Exports `formatAmount(amountAic, { currency, precision })`: AIC short = `Math.ceil(amount) + ' AIC'`; AIC full = `amount.toFixed(2) + ' AIC'`; USD short = `'$' + (Math.ceil(amountAic) / 100).toFixed(2)`; USD full = `'$' + (amountAic / 100).toFixed(2)`. Currency comes from `getDisplayCurrency()`. No `mode` field, no tilde decoration — values are measured. Used by editor surfaces only; the trailer writer in `trackingFile.ts` formats numbers directly without this helper.
- **statusBar.ts** — Status bar item showing total cost via `formatAmount(..., precision: 'short')`. Tooltip shows total + per-model rows via `formatAmount(..., precision: 'full')` with no disclosure line. `createStatusBar` returns a handle with `setNudge(visible: boolean)`: when `visible` is true the item renders `$(refresh) Copilot Budget — reload to start tracking` bound to `workbench.action.reloadWindow`; when false it renders the normal cost bound to `copilot-budget.showStats`. Re-renders on `onConfigChanged` for `copilot-budget.displayCurrency` (so the unit flips without waiting for the next 30s poll).
- **budgetPanel.ts** — `showBudgetPanel({ tracker })` renders the Copilot Budget panel as a VS Code QuickPick. Two toggle rows at the top (currency, commit hook), separator, total + per-model breakdown, separator, Refresh. The OTel-enable row is gone — auto-enable handles it. Currency/hook state is re-read fresh on every render via `tracker.getStats()`, `getDisplayCurrency()`, `isHookInstalled()`.
- **trackingFile.ts** — Reads and writes stats to `<gitdir>/copilot-budget` in key=value format for the commit hook to read. The pure `formatTrackingFile(stats: TrackingStats): string` builds the file contents (newline-terminated); `writeTrackingFile()` is a thin async wrapper that resolves the uri via `resolveGitDir` and writes via `writeTextFile` from `fsUtils`. Splitting these lets tests (e.g. `hook-git-e2e.test.ts`) format the file with synthetic stats and write it through real Node `fs` instead of the `vscode.workspace.fs` mock. Each worktree gets its own tracking file. Schema: `SINCE`, `INTERACTIONS`, `TOTAL_AI_CREDITS`, per-model `MODEL_<sanitized>_{INPUT,OUTPUT,CACHE_READ,CACHE_CREATION}_TOKENS` plus `_COST_AIC`, and `TR_`-prefixed trailer lines. No `MODE=` line is written. All `TR_` trailer values are bare two-decimal numbers (no `~`, no `$`) — the trailer name conveys the unit. The `_COST_AIC` machine-readable keys stay unprefixed. Per-model AI Credits trailer uses `getDisplayName()` from `tokenRates.ts`. When the opt-in `estimatedCost` trailer is enabled, the USD value is derived inline from `totalAiCredits / 100` at trailer-write time. `parseTrackingFileContent(content)` requires at least `SINCE` and `TOTAL_AI_CREDITS`; legacy v0.6.x files containing `TOTAL_COST_USD` / `_COST_USD` keys and legacy v2.0.x files containing `MODE=files` / `MODE=telemetry` are tolerated by silently ignoring those keys. Pre-0.6 files (no `TOTAL_AI_CREDITS`) return `null`.
- **fsUtils.ts** — Thin wrappers around `vscode.workspace.fs` to avoid repetitive TextEncoder/TextDecoder boilerplate. Exports `readTextFile(uri)`, `writeTextFile(uri, content)`, and `stat(uri)`. All return promises and handle errors gracefully (returning `null` on failure). Used by `gitDir.ts`, `trackingFile.ts`, and `commitHook.ts`.
- **gitDir.ts** — Shared utility for resolving git directories. `resolveGitDir(workspaceRoot: vscode.Uri): Promise<vscode.Uri | null>` follows `.git` files (worktrees, submodules, devcontainers) to the real git dir. `resolveGitCommonDir(workspaceRoot: vscode.Uri): Promise<vscode.Uri | null>` further follows the `commondir` file in worktrees to reach the shared git dir. `resolveHooksDir(workspaceRoot: vscode.Uri): Promise<vscode.Uri | null>` returns where git will look for hook scripts: it first consults `core.hooksPath` via `getGitConfigValue(cwd, 'core.hooksPath')` (spawns `git -C <cwd> config --get core.hooksPath` with a 3s timeout, returns `null` on any error including ENOENT and exit 1), resolves absolute paths verbatim and relative paths against the worktree root (matching git's own behavior), and falls back to `<gitCommonDir>/hooks` when unset. This makes the hook installer respect husky/lefthook/custom `core.hooksPath` configurations. Uses `fsUtils` helpers for filesystem access via `vscode.workspace.fs`.
- **commitHook.ts** — Installs/uninstalls a POSIX `prepare-commit-msg` hook that reads `copilot-budget` from the git dir and appends git trailers to commit messages. The hook uses a generic `TR_` line protocol: if the tracking file contains any `TR_<name>=<value>` lines, the hook converts them to git trailers via sed (e.g., `TR_Copilot-AI-Credits=42.31` becomes `Copilot-AI-Credits: 42.31`); otherwise it skips. Trailer names are configured in `config.ts` and written by `trackingFile.ts`. All exported functions (`isHookInstalled`, `installHook`, `uninstallHook`) are async. `getHookUri()` calls `resolveHooksDir` (from `gitDir.ts`) so the install path follows `core.hooksPath` when set (husky's `.husky/`, lefthook, custom shared hook dirs) — install/uninstall/isInstalled all stay aligned with where git actually looks. Uses `readTextFile`/`writeTextFile` from `fsUtils`. The installer is also a refresher: when the existing hook at the resolved path contains the `MARKER` line it is overwritten silently with the current `HOOK_SCRIPT` (so upgrading the extension auto-replaces stale hook scripts on the next activation); when it contains anything else, install aborts with a warning rather than clobbering a third-party hook. Includes a `makeExecutable` helper that uses `fs.chmodSync` for local files and a VS Code shell task for remote filesystems. The hook script uses `git rev-parse --git-dir` at runtime for worktree-safe tracking file resolution. Resets the tracking file after appending (only on the normal-commit path). On rebase (detected via `$GIT_DIR/rebase-merge` or `$GIT_DIR/rebase-apply` — note: `git rebase -i` with squash fires the hook with `$2 == message`, NOT `squash`) or on `$2 == squash` (which is `git merge --squash` followed by `git commit` *without* `-m`, since `-m` sets `$2 == message`), the hook sums duplicate trailer lines whose value is a bare number — in practice the default `Copilot-AI-Credits:` total. The opt-in aggregate per-model trailer (`Copilot-AI-Credits-Models: A=N,B=M`) has a non-numeric value and is not matched; the `Copilot-Est-Cost` trailer and any user-renamed `aiCredits` key are also not matched (the awk regex hard-codes `Copilot-AI-Credits`). The sum path does NOT consult the tracking file or reset it, so usage accumulated during a rebase carries forward to the next normal commit. `git cherry-pick` falls through to the normal-commit path (not the sum path), so cherry-picking a commit that already carries a trailer while the local tracking file ALSO has stats results in two `Copilot-AI-Credits:` lines on the new commit — the hook appends rather than summing here. `git revert` also falls through to the normal-commit path, but git's default revert message (`Revert "<subject>" / This reverts commit <sha>.`) does NOT inline the reverted commit's body, so no inherited trailer reaches the new message file; with no local tracking file the revert commit ends up with zero `Copilot-AI-Credits:` lines.
- **config.ts** — Wraps `copilot-budget.*` settings plus the upstream `github.copilot.chat.otel.dbSpanExporter.enabled` read/write paths. Exports `TrailerConfig` interface and `getTrailerConfig()`, `getDisplayCurrency(): 'aic' | 'usd'`, `isOTelDbExporterEnabled(): boolean`, `autoEnableOTel(): Promise<void>` (strictly-asymmetric write: if `inspect(OTEL_KEY)` shows both `globalValue` and `workspaceValue` are `undefined`, write `true` at Workspace scope; never write `false`; failures are logged but never thrown so a transient settings-write error cannot block activation), and the OTel section/key constants. Defaults: `aiCredits` → `"Copilot-AI-Credits"` (on; primary trailer), `estimatedCost` → `false` (opt-in USD trailer), `aiCreditsPerModel` → `false` (opt-in), `displayCurrency` → `"aic"`.
- **utils.ts** — Shared utility helpers. Exports `sanitizeModelName(name)` (replaces non-alphanumeric characters with underscores for key=value tracking file keys) and `errorMessage(err)` (safely extracts a message string from unknown thrown values). Used by `tracker.ts`, `trackingFile.ts`, and `commitHook.ts`.
- **logger.ts** — Shared OutputChannel logger singleton. Exports `log()` (timestamped append), `getOutputChannel()`, and `disposeLogger()`. Used by sessionDiscovery, tracker, config (`autoEnableOTel`), and the diagnostics command.

## Testing

Tests live alongside source files as `*.test.ts`. The `vscode` module is mocked via `src/__mocks__/vscode.ts` (includes `workspace.fs`, `Uri.joinPath`, `FileType` enum, `Task`/`ShellExecution` mocks, `ConfigurationTarget` enum, `QuickPickItemKind`, and a `__workspaceUpdate` `jest.fn` capturing every `getConfiguration(section).update(key, value, target)` call for asymmetric-write assertions). Tests for workspace-side modules (`gitDir`, `trackingFile`, `commitHook`) mock `./fsUtils` rather than Node.js `fs`. Session discovery tests still mock `fs` directly. `otelReader.test.ts` synthesizes a fixture SQLite DB in `beforeAll` via real `node:sqlite` (covering `aggregateSince` with session-id scoping, parent-session OR-join for title subagents, cache_creation summed from `span_attributes`, orphan exclusion, the retention edge case, and zero-session-ids early-return) — no checked-in binary.

`hook-git-e2e.test.ts` is the only suite that drives real `git` against throwaway repos created via `fs.mkdtempSync`. It's gated on `gitAvailable()` (`execFileSync('git', ['--version'])`) and uses `describe.skip` + `console.warn` when git is absent. The harness env block forces `HOME=<tmpdir>`, `GIT_CONFIG_GLOBAL=/dev/null`, `GIT_CONFIG_SYSTEM=/dev/null`, `GIT_EDITOR=true`, and explicit `GIT_AUTHOR_*`/`GIT_COMMITTER_*`, and scrubs `GIT_DIR`/`GIT_WORK_TREE`/`GIT_INDEX_FILE`/`GIT_TEMPLATE_DIR` — never touches the user's `~/.gitconfig` or CI's `/etc/gitconfig`. `hook-script.test.ts` remains as the fast shell-only suite that synthesizes `.git/rebase-merge/` by hand; the two are complementary and the overlap is intentional.

## Key Design Details

- **Auto-enable upstream OTel** — `autoEnableOTel()` runs once at activation. It calls `inspect(OTEL_KEY)` and writes `true` at `ConfigurationTarget.Workspace` only when both `globalValue` and `workspaceValue` are `undefined`. An explicit user choice at either scope is respected. The write is strictly asymmetric: only unset → `true`, never the reverse — disabling the upstream exporter requires the user to flip the setting via VS Code Settings. The helper catches and logs all errors so a transient settings-write failure does not block activation.
- **Status-bar nudge** — after auto-enable, Copilot Chat typically needs a reload before it begins writing spans. When `isOTelDbExporterEnabled()` is `true` AND `reader.isAvailable()` is `false`, `statusBar.setNudge(true)` swaps the item to `$(refresh) Copilot Budget — reload to start tracking` bound to `workbench.action.reloadWindow`. The `onStatsChanged` handler polls `reader.isAvailable()` on every poll cycle and calls `setNudge(false)` exactly once when the DB shows up, then unhooks (`nudgeCleared` flag).
- **Window-scoped discovery** — each VS Code window scans only its own `workspaceStorage/<hash>/GitHub.copilot-chat/transcripts/` plus the legacy `chatSessions/` directory, and uses the resulting UUID stems as the OTel SQL session-id filter. Background "title" subagent spans (gpt-4o-mini) are attributed via the `parent_chat_session_id` OR-clause in the same query. Same-repo dual-window remains last-writer-wins on the tracking file (shared hash → shared discovery → shared tracking file path); pre-existing limitation, out of scope.
- **Remote-host limitation** — `agent-traces.db` lives where Copilot Chat runs (typically workspace-side in remote dev). When the upstream setting is `true` but the DB is missing locally, the reader's `isAvailable()` returns `false` so the status-bar nudge fires. `extensionKind` is `["workspace","ui"]` so VS Code prefers the workspace install in remote contexts (SSH, WSL, devcontainers, Codespaces) — where both transcripts and the OTel DB live workspace-side — while still falling back to `ui` for empty-window activation (no workspace ⇒ no "workspace" target ⇒ UI host runs the extension).
- **No runtime YAML dependency** — `js-yaml` is dev-only. The build step in `esbuild.js` parses `data/models-and-pricing.yml` and writes `dist/models-and-pricing.json`; the runtime loader (`tokenRates.ts`) uses `JSON.parse`. Tests generate `src/__fixtures__/models-and-pricing.json` from the YAML fixture via `jest.globalSetup.js`.
- **Workspace-side file I/O uses `vscode.workspace.fs`** — Modules that access workspace files (`gitDir.ts`, `trackingFile.ts`, `commitHook.ts`) use `vscode.workspace.fs` via `fsUtils.ts` wrappers, enabling transparent remote filesystem proxying in devcontainers, Codespaces, and SSH Remote. Only `commitHook.ts` retains a Node.js `fs` import for local `chmodSync`. The host-side modules `sessionDiscovery.ts` and `otelReader.ts` use Node.js `fs` / `node:sqlite` since they read from the host machine's filesystem.
- **esbuild bundles** to a single `dist/extension.js` (CommonJS, **Node 22 target** — required for `node:sqlite`, vscode external). The build also converts `data/models-and-pricing.yml` to `dist/models-and-pricing.json`. `engines.vscode` is pinned to `^1.103.0` (first release bundling Electron 37 / Node 22.17.0, where `node:sqlite` is stable); pre-1.103 VS Code installs cannot load the extension.
- The commit hook is pure POSIX shell with no external dependencies.
- The extension makes no network calls at runtime. All session data is read from a local SQLite DB; the rate card is shipped in the extension bundle and refreshed by contributors via `npm run update-rates`. Only aggregate token counts are read from OTel spans — prompt and response content are never touched.
